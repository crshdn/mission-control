acceptance_criteria: 
1. The `crusty-cutline` wrapper must present a clear, structured preview of all parsed fields before writing to Mission Control.
2. The intake script must enforce a readiness gate for "task" mode, blocking `/confirm` if required fields (like acceptance_criteria and non_goals) are missing.
3. Users must be able to inject missing fields via conversational replies in Telegram, which update the active draft.
4. The `/cancel` command must cleanly wipe the local draft state without creating orphaned records in the Mission Control database.
5. The final `/confirm` must successfully push the completed object to the Mission Control API and generate the local Obsidian vault note.

non_goals: 
1. We are not building a complex conversational state machine or multi-step interactive wizard in Telegram.
2. We are not modifying the core Mission Control database schema or API endpoints; all changes must be contained within the intake scripts (`cutline-telegram-intake.ts` and the `crusty-cutline` wrapper).
3. We are not adding support for non-build lanes (Signal, Line Lab) to this specific refinement flow yet.
